\begin{abstract}
A growing class of optimization problems are tackled by an LLM \emph{agent} that proposes a candidate (a code edit, a hyperparameter assignment, a data-mix vector), evaluates it via a fixed harness, and iterates. Two extremes dominate today's implementations: \emph{serial loops} that use information optimally per decision but waste compute (one GPU evaluating, $K-1$ idle), and \emph{naive parallel loops} that scale compute but starve each decision of the most recent results. We characterize this as a tradeoff between two scarce resources — \emph{information per decision} and \emph{parallel throughput} — and propose a \textbf{resource-aware} framework that exposes the tradeoff as three independent knobs. The framework introduces a manager-issued cumulative-lineage primitive that selectively re-shares wins across parallel workers, a watermark-gated re-planning policy that controls the manager's wake-up rate, and a compact rationale ledger that re-injects each evaluation's hypothesis next to its outcome on every wake-up. We evaluate on two task plugins — a hyperparameter-search task on a small GPT trainer, and a Qwen-3 SFT data-mix-search task on T\"ulu-3 — and show the resource-aware configuration recovers most of the serial loop's research output while keeping $8\times$ throughput.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

A growing fraction of practical optimization is now done by an LLM \emph{agent} in a loop: the agent reads prior results, proposes a candidate (a code edit, a hyperparameter setting, a configuration), evaluates it via a fixed harness (a training run, an eval, a benchmark), reads the metric, and iterates. The same pattern recurs across automated machine-learning research \cite{karpathy2026autoresearch}, code-fix loops in software engineering agents \cite{aider, swebench}, prompt-tuning loops, kernel-autotuning loops, and data-mix-search loops in instruction-tuning \cite{tulu3}. We call this class \emph{agent-led search}. It is distinguished from classical AutoML by the proposer's nature: the proposer is a generative LLM with prior domain knowledge, free-form access to the search space, and rationales that themselves enter the next iteration's history.

Two implementation extremes dominate today:

\textbf{Serial loops} run one agent in a single repository, evaluating one candidate at a time. Each iteration sees the full history — including the just-finished result — and the agent's git branch advances on every win. Serial loops use available information \emph{optimally}: every byte of state at iteration $t$ informs the proposal at $t{+}1$. They are wasteful of compute: on an 8-GPU node, 7 GPUs idle through every evaluation.

\textbf{Naive parallel loops} ask a manager agent to dispatch a batch of $K$ candidates that workers run in parallel; the manager reads the batch and dispatches the next. Compute is fully utilized, but information is fragmented: each batch's $K$ proposals are committed before any of their results land; the next batch reflects only the previous batch's outcomes; per-result reactions are coarsened to per-batch reactions. We empirically observed this in our setting: an 8-GPU 1-hour run produced 27 evaluations whose best result fell within seed-noise of the starting baseline, despite a competent manager LLM and clean infrastructure.

The two extremes optimize different resources. The serial loop spends \emph{information per decision} liberally and \emph{parallel throughput} not at all; the naive parallel loop does the reverse. Neither is satisfying for a multi-GPU shared allocation, which is the dominant deployment context for the manager-worker pattern. The honest design space lies between them: \emph{which pieces of information actually matter, and how much do we lose if we let them lag?}

This paper makes three contributions:

\begin{enumerate}[leftmargin=*,itemsep=2pt]
  \item \textbf{Tradeoff characterization} (Section~\ref{sec:method}). We identify three concrete information-flow channels — state, baseline, and coverage — that are silently broken by naive parallelization, and analyze each independently.

  \item \textbf{Resource-aware framework} (Section~\ref{sec:system}). We introduce three matching mechanisms — a watermark-gated re-planning policy, a cumulative-lineage primitive (\texttt{KEEP\_EXPERIMENT}), and a compact rationale ledger with verbatim source-file injection — that selectively restore each channel without giving up parallelism. Each mechanism has a parameterized cost and can be enabled independently.

  \item \textbf{Empirical evaluation on two tasks} (Section~\ref{sec:experiments}). We evaluate on a hyperparameter-search task and a Qwen-3 SFT data-mix-search task at matched 1-hour wall budgets against both serial and naive-parallel baselines, with per-channel ablations. The resource-aware configuration produces an order of magnitude more useful signal than naive parallel and approaches the serial baseline at $8\times$ throughput.
\end{enumerate}

The framework is task-agnostic; tasks plug in via a five-file contract.

\section{Related Work}
\label{sec:related}

\subsection{Agent-led research and engineering loops}

Karpathy's autoresearch \cite{karpathy2026autoresearch} is the most direct antecedent and the source of our serial baseline. Its design is the canonical serial loop: one agent in a single git repository, fixed-budget evaluations, branch-advance on win. The strength is information utilization; the limit is parallelism.

Multi-agent code-editing systems such as Aider \cite{aider} and the systems benchmarked on SWE-bench \cite{swebench} parallelize work across files or sub-tasks but do not have a primitive for cumulative branching across the parallel pool — each parallel agent's edits typically land on a per-task branch and are merged by a coordinator at completion. AutoGPT-style loops dispatch sub-goals in parallel without a shared baseline-advance mechanism; they suffer the analogous baseline-staleness problem in our taxonomy.

Tree-search planners such as LATS \cite{lats} and ReAct \cite{react} expand parallel branches in an explicit tree with state-merging policies; they assume a stateful environment a tree-search node can be checkpointed to, while LLM-driven research loops have side-effecting code edits we leverage \texttt{git} to manage.

\subsection{Bayesian and population-based hyperparameter search}

Population-based training \cite{pbt} and Hyperband \cite{hyperband} share the structure of parallel evaluations with periodic exploitation, but their proposal mechanism is restricted (perturbation, halving). Bayesian optimization assumes a posterior over a low-dimensional continuous search space. The agent-led setting differs because the proposer has free-form generative access to the search space and uses prior history qualitatively, not just quantitatively. Our framework is complementary to BO/population-based methods: a future variant could let the manager LLM act as a high-recall proposer with a learned surrogate ranking candidates before dispatch.

\subsection{Data-mix search for instruction tuning}

T\"ulu-3 \cite{tulu3} curates per-source labels in its mixture, which is what makes our second task have signal. Recent work on automated mix-weight tuning \cite{datamix2024} is parallel and population-based. The agent-led approach is complementary, and our framework's task-plugin contract lets the same scheduler / lineage / ledger machinery host either a hyperparameter search or a data-mix search by swapping the runnable code and the manager prompt.

\section{Method: The Information-Parallelism Tradeoff}
\label{sec:method}

\subsection{Problem setup}
\label{sec:setup}

A search instance is a tuple $(\mathcal{S}, h, \pi)$ where:

\begin{itemize}[leftmargin=*,nosep]
  \item $\mathcal{S}$ is the search space: legal mutations to a fixed source repository.
  \item $h: \mathcal{S} \to \mathbb{R}$ is a deterministic harness mapping a candidate to a metric, with fixed wall-clock cost $T_h$ per call.
  \item $\pi$ is the search policy: an LLM agent that, given a history $H_t = \{(s_i, h(s_i), r_i)\}_{i \le t}$ (where $r_i$ is the rationale produced when proposing $s_i$), emits the next candidate $s_{t+1}$.
\end{itemize}

The objective is $\min_{s \in \mathcal{S}_{\text{visited}}} h(s)$ given a wall-clock budget $W$ and a parallel-evaluation budget $K$ (typically the GPU count). Two limiting regimes:

\begin{itemize}[leftmargin=*,nosep]
  \item $K{=}1$ \textbf{(serial):} $\pi$ sees the full $H_t$ before each $s_{t+1}$. Total evaluations $N_W = W / T_h$.
  \item $K{=}K_{\max}$ \textbf{(naive parallel):} $\pi$ emits $K$ candidates per wake-up, sees only $H_t$ at the wake-up time. Total evaluations $N_W \approx K \cdot W / T_h$ but each batch reflects an information lag of up to $T_h$.
\end{itemize}

We seek a policy that approaches naive parallel's $K \cdot W / T_h$ throughput while approaching serial's information utilization.

\subsection{Three information-flow channels}

Three concrete information-flow channels distinguish a serial loop from a naive parallel one. Each is independent — a parallel framework can restore some without restoring all.

\textbf{Channel 1: State staleness.} A serial loop calls $\pi$ exactly once per evaluation. A naive parallel loop calls $\pi$ once per \emph{batch}, so the proposal $s_{t+1}$ is conditioned on history that lags by up to $K-1$ evaluations. If the agent's reasoning depends on individual results (``the $\eta=0.05$ result tells me to try $0.045$ next''), batching delays this by $T_h$ wall-time. The mitigation cost is \emph{agent LLM calls}: calling $\pi$ on every individual completion scales agent cost by $K$.

\textbf{Channel 2: Baseline staleness.} A serial loop's git branch advances on every win: evaluation $t{+}1$ runs from a source state that includes evaluation $t$'s edit, if $t$ won. Wins compound — eight serial wins produce a candidate with eight cumulative edits applied. A naive parallel loop forks each evaluation from a fixed baseline; wins do not compound across the batch. The mitigation cost is \emph{state coherence}: in-flight evaluations dispatched before a baseline advance are now ``stale.'' We must decide whether to cancel and re-dispatch (lost compute), let them complete and discard (lost compute), or let them complete and accept the staleness (engineering complexity).

\textbf{Channel 3: Coverage staleness.} A serial loop sees a single linear history. The agent reads prior \texttt{(edit, outcome)} pairs and naturally tracks which axes it has tried at which values. A parallel loop's manager sees compact summaries. We empirically observe a failure mode we call \emph{axis fixation}: once the manager finds a winning axis (e.g., $\texttt{aspect\_ratio}$), subsequent waves dispatch a long tail of \texttt{aspect\_ratio\_X} combinations while other axes go untouched. In one of our autoresearch runs, 40 of 65 dispatches were variants on a single axis. The mitigation cost is \emph{prompt context length}: explicit coverage tracking added to the manager's per-cycle context.

\section{System Design}
\label{sec:system}

We instantiate the method as a single Node.js process per allocation, holding state in a cached SQLite handle (WAL) for token grants, JSON event-trace files (one per evaluation) for the rationale ledger, and the per-task source repository's \texttt{.git} for the cumulative lineage chain. The framework is $\sim$5{,}000 lines across nine modules. We use Claude Opus or Codex CLI as the manager; either accepts the same TASK\_GRAPH protocol.

\subsection{Watermark-gated re-planning (Channel 1)}
\label{sec:watermark}

We replace ``call manager once per batch'' (naive parallel) and ``call manager once per evaluation'' (serial-ish) with a parameterized policy. Let $N_{\text{ready}}$ and $N_{\text{running}}$ denote the queue depth and in-flight count at any moment. We wake the manager only when

\begin{equation}
N_{\text{ready}} + N_{\text{running}} < \alpha \cdot N_{\text{workers}}
\label{eq:watermark}
\end{equation}

where $\alpha \in (0, \infty)$ is a per-task knob. Three regimes:

\begin{itemize}[leftmargin=*,nosep]
  \item $\alpha = 1.0$: wake on every completion (serial-style; expensive).
  \item $\alpha = 1.5$: wake when queue depth drops below $1.5 K$ (naive-parallel-style; cheap).
  \item $\alpha < 1.0$: aggressive — wake before completions, useful for cold-start.
\end{itemize}

A minimum-interval seconds parameter prevents wake-up storms when many workers complete near-simultaneously. Completed tasks are queued FIFO into the next wake-up regardless. The trade-off is direct: $\alpha$ controls agent LLM call count vs.\ per-result reactivity.

\subsection{Cumulative lineage via \texttt{KEEP\_EXPERIMENT} (Channel 2)}
\label{sec:lineage}

We introduce a manager-issued primitive that explicitly advances the shared source baseline. The manager emits, alongside its TASK\_GRAPH:

\begin{lstlisting}
<!-- KEEP_EXPERIMENT -->
[{"id": "exp_0042_aspect_80",
  "reason": "aspect=80 -0.0014 vs HEAD"}]
<!-- /KEEP_EXPERIMENT -->
\end{lstlisting}

The runtime, on receiving this directive: (1) looks up the experiment's structured edits from its task-events trace; (2) re-applies them to the shared source repository via the same edit-application pipeline used for per-evaluation sandboxes; (3) commits with the manager's reason; (4) appends a row to a \texttt{lineage.json} log (\texttt{\{experiment\_id, commit, reason, kept\_at\}}). Subsequent evaluations clone from the now-advanced source.

\textbf{Race handling.} An evaluation dispatched with the old baseline before a KEEP fires runs to completion as if the KEEP never happened. We deliberately do \emph{not} cancel it — the in-flight result is still informative (it tells the manager what the previous baseline yielded). The manager observes both the kept lineage and the stale-baseline result on its next wake-up and decides whether to KEEP the stale-baseline result on top, kill it, or ignore it. Lineage is an explicit manager decision, never a timing artifact.

\textbf{Cost trade.} The mitigation cost is bounded: even if all $W$ in-flight evaluations dispatched before a KEEP turn out to be ``wasted,'' the lag cost is one wall-cycle, after which the next batch dispatches from the new baseline. Empirically, in-flight evaluations from the old baseline frequently \emph{are} useful — they include hypotheses orthogonal to the kept axis, which the manager can then KEEP on top.

\subsection{Compact rationale ledger (Channel 3)}
\label{sec:ledger}

We restore code-reading and history-reading via two task-agnostic injections into the manager's per-cycle context:

\textbf{Verbatim source files.} The task plugin declares a \texttt{managerContextFiles} list (e.g., \texttt{[train.py]}). On every manager wake-up, the runtime reads those files from the current source HEAD and injects them inline. The manager sees the current code (post-KEEP) verbatim. This is what allows the manager to notice e.g.\ ``\texttt{COMPILE\_DEFAULT="off"} is suspicious; let me try flipping it'' — a common-sense observation that ledger summaries don't surface but the serial agent gets ``for free'' by virtue of being in the repo.

\textbf{Rationale ledger.} For each completed evaluation, the runtime records the manager's hypothesis (``\textit{wider aspect ratio matches FFN dim better}'') alongside the structured edit (\texttt{ASPECT\_RATIO 64$\rightarrow$96}) and the outcome (\texttt{val\_bpb=0.9928}). On every wake-up, the top-K and most-recent rows are re-injected as a compact list. The manager sees its own past reasoning chain — the analog of a serial agent's accumulated experiment notes. We make the rationale field \emph{mandatory} on every dispatched task; without it, the next wake-up cannot distinguish ``aspect=80 won by signal'' from ``aspect=80 won by luck.''

\textbf{What we do not yet implement.} A coverage-tracking section (\textit{axes\_explored: matrix\_lr was tried at \{0.03, 0.06, 0.08\} — 0.05 unsampled}) would directly mitigate axis fixation by aggregating from the picked-event stream. We observe the failure mode and discuss this mitigation in Section~\ref{sec:discussion}.

\subsection{Two execution modes per task}

The TASK\_GRAPH schema accepts \texttt{execution\_mode} per task. \texttt{param\_patch} (the default) supplies a structured \texttt{edits[]} array that the framework applies deterministically — no per-evaluation LLM call. Edits come in four kinds: \texttt{constant\_replace} (module-level Python assignment, with AST-normalized comparison so $2^{19}$, \texttt{524288}, and \texttt{1<<19} all match), \texttt{regex\_replace} with a mandatory \texttt{expected\_count} that fails loudly on drift, \texttt{block\_replace} between anchor regexes, and \texttt{unified\_diff} as escape hatch. \texttt{code\_edit} falls back to an LLM worker for genuinely free-form rewrites (architecture restructures, optimizer swaps). In our experiments $\geq 90\%$ of dispatched tasks ran via \texttt{param\_patch}, eliminating per-task LLM cost.

\subsection{Task plugin contract}

A new task plugs in via five files: (1) runnable code in a directory, (2) a bash wrapper that runs one evaluation and writes \texttt{result.txt} (\texttt{exit\_code}) plus \texttt{metrics.json} (the metric to optimize), (3) a manager prompt describing the search surface, (4) a worker prompt for the LLM-driven fallback path, and (5) a workspace generator that materializes per-run config. The framework supplies the scheduler, dispatch primitives, lineage management, ledger, and timeouts; none is task-specific.

\subsection{Engineering details}

We briefly note three engineering decisions that affect reproducibility but not the conceptual contribution. \emph{Walltime admission}: tasks with \texttt{remaining\_wall < estimated\_runtime + slack} are blocked, not started. \emph{Three concentric timeout rings}: in-wrapper, orchestrator hard-cap (\texttt{SIGTERM} via process group, \texttt{detached: true} spawn), and manager-supplied per-task early-stop on intermediate metrics. \emph{Persistent shared compile cache}: \texttt{torch.compile}'s inductor + Triton kernel cache is shared across evaluations and across batch jobs, so cold-compile is amortized.

\section{Experiments}
\label{sec:experiments}

\subsection{Configurations}

We compare three configurations on each of two task plugins. All comparisons hold wall-clock budget fixed at 1 hour; the parallelism budget differs by configuration.

\begin{itemize}[leftmargin=*,nosep]
  \item \textbf{Serial} ($K{=}1$). Single-agent loop, no framework involvement: a Claude Opus session in \texttt{--print} mode with bash and edit access, operating in a fresh clone of the task source. Follows the task's \texttt{program.md} (Karpathy's, in the autoresearch case). The agent advances or rewinds its git branch on each result.
  \item \textbf{Naive parallel} ($K{=}8$). Our framework with $\alpha{=}1.5$ watermark and cumulative lineage \emph{disabled}. Each evaluation forks from a fixed baseline; the manager has no \texttt{KEEP\_EXPERIMENT} primitive.
  \item \textbf{Resource-aware} ($K{=}8$). Our framework with $\alpha{=}1.0$, cumulative lineage enabled, rationale ledger and source-file injection enabled.
\end{itemize}

All configurations use Claude Opus as the agent / manager. All share the same persistent \texttt{torch.compile} cache, so cold-compile costs are comparable in the first cycle of any configuration.

\subsection{Tasks}

\textbf{Task A: hyperparameter search (autoresearch).} A small GPT trainer; the agent edits \texttt{train.py}; each evaluation trains for a fixed 5-minute compute budget; the metric is \texttt{val\_bpb} on a held-out evaluation set, lower is better. The search space is open: model width, depth, head dim, attention pattern, optimizer LR, batch size, schedule shape. The task is Karpathy's autoresearch \cite{karpathy2026autoresearch}.

\textbf{Task B: SFT data-mix search (qwen-sft).} Full fine-tuning of Qwen-3-0.6B on a weighted mixture of T\"ulu-3 instruction-following data, bucketed into five domains: \{math, code, chat, instruction-following, reasoning\}. The agent edits a single \texttt{DATA\_MIX} dict in \texttt{train.py}, holding everything else constant. Each evaluation does 10-minute SFT and reports balanced held-out val\_loss across the five domains, lower is better. Naturally domain-labeled data is what makes this task have signal — the agent searches \emph{which mixture weights are right}, not which seed performed best.

Three observations from Table~\ref{tab:auto}.

\textbf{Naive parallel does not convert throughput into improvement.} 27 evaluations all clustered near the starting baseline. We sampled the manager's rationales and found them sensible (e.g., ``aspect=96 should better match FFN dim''), so the failure was not in proposal quality. The failure was in baseline staleness — every evaluation forked from the same un-advanced source.

\textbf{Resource-aware recovers most of the serial loop's signal.} A $-0.0010$ improvement is an order of magnitude larger than naive-parallel's $-0.0001$. The single \texttt{KEEP\_EXPERIMENT} block that fired stacked three independent winners (aspect=80, final\_lr\_frac=0.05, warmdown=0.55) — exactly the cumulative-branch behavior the serial loop achieves naturally.

\textbf{Resource-aware is still slightly behind serial} at this wall budget. Inspection shows the manager only emitted one \texttt{KEEP\_EXPERIMENT} block in the entire hour, and it fired at $t \approx 55$ minutes — leaving $\sim$5 minutes for follow-on evaluations to exploit the advanced HEAD. We discuss this in Section~\ref{sec:discussion}.

\subsection{Costs}

The resource-aware configuration's mitigation costs are bounded:

\begin{itemize}[leftmargin=*,nosep]
  \item \textbf{Manager LLM calls.} 14 in 1 hour at $\alpha{=}1.0$, vs.\ $\sim$10 at $\alpha{=}1.5$ — a 40\% increase. Per-completion cost (manager calls / completions) was 0.22:1, roughly $4.5\times$ fewer calls than serial would issue at the same throughput.
  \item \textbf{In-flight invalidation.} 7 evaluations were in flight when the lone KEEP fired; all 7 completed and were observed by the manager (none cancelled).
  \item \textbf{Prompt token cost.} Source-file injection added $\sim$3 KB per wake-up; ledger adds another $\sim$2 KB. Negligible against modern context windows.
\end{itemize}

\section{Discussion and Open Problems}
\label{sec:discussion}

\textbf{KEEP rate is a research bottleneck.} The manager fired exactly one \texttt{KEEP\_EXPERIMENT} block in 60 minutes. Examination of its responses suggests two factors: it interpreted the prompt's noise-tolerance hint conservatively (waiting for clearer wins), and batched dispatch let it react to multiple results at once, suppressing per-result KEEPs. Whichever factor dominates, the bulk of the parallel budget runs on a single un-advanced baseline. We argue this is a prompt-engineering and sequential-decision-theory question: a noise-aware threshold (KEEP iff $\Delta$metric $> k\sigma_{\text{noise}}$, with $k$ derived from remaining wall and $K$) would directly address it. Estimating $\sigma_{\text{noise}}$ requires multi-seed re-evaluation of representative points; we sketch this as future work.

\textbf{Axis fixation and coverage steering.} We described the axis-fixation failure mode in Section~\ref{sec:method} and observed it empirically (40 of 65 evaluations on a single axis). We do not yet mitigate it. A simple programmatic remedy is to extract the (axis, value) pair tested from each completed evaluation's structured edits, aggregate into a manifest, and inject into the manager's per-cycle context: \textit{``axes\_explored: matrix\_lr in $\{0.03, 0.06, 0.08\}$; 0.05 is unsampled interior; weight\_decay never tested.''} This is an additive, no-risk change. We expect it to materially shift the resource-aware result toward the serial baseline.

\textbf{Generality.} The three channels (state, baseline, coverage) and their mitigations (watermark, lineage, ledger) are not specific to either task we evaluate. They generalize to any optimization problem of the form $\arg\min_s h(s)$ where the proposer is an LLM with access to history, the harness is a black box, and parallel evaluation is available. We expect the framework to translate to code-improvement loops (where $h$ is unit-test pass rate), prompt search (where $h$ is held-out task accuracy), kernel autotuning (where $h$ is throughput), and architecture search outside language modeling. Each requires only the five-file plugin contract; the framework's primitives carry over.

\section{Conclusion}
\label{sec:conclusion}

Agent-led search occupies a distinctive point in the optimization-policy design space: the proposer is an LLM with generative access to the search space and history, and the harness is an expensive black box. Serial loops and naive parallel loops dominate today's implementations and leave a wide tradeoff frontier untouched. We characterize the tradeoff as three independent information-flow channels (state, baseline, coverage) and propose a resource-aware framework that exposes each as a knob or primitive. Cumulative lineage via \texttt{KEEP\_EXPERIMENT} restores baseline coherence; watermark-gated re-planning restores per-result reactivity at parameterized cost; a compact rationale ledger plus verbatim source-file injection restore code-reading and history-reading. Together they recover most of a serial loop's research output at $8\times$ throughput. The framework is open-source; both task plugins ship with it.

\section{Implementation Details}
\label{app:impl}

\textbf{Spawn isolation.} Each parallel evaluation spawns its harness wrapper in a detached process group with cwd set to a per-evaluation sandbox directory cloned from the task source repository. The wrapper environment receives an explicit \texttt{REPO\_ROOT} pointing to the sandbox. This is non-trivial: an early version had a bug where the wrapper hard-coded \texttt{repo\_root="\$script\_dir/source"} and silently ignored the cwd, causing all parallel evaluations to read the same shared source. The symptom was 27 evaluations clustered within seed-noise; the cause was a single missing environment variable.

\textbf{Drain probes.} After SIGKILL of a runaway evaluation, we briefly poll \texttt{nvidia-smi --query-compute-apps} before reissuing the freed GPU token, to catch process-leak edge cases where the kernel reaper races CUDA context release.

\textbf{Filesystem locking.} The cumulative-lineage \texttt{git commit} on the shared source must protect against \texttt{git add --all} timeouts on slow network filesystems; we observed 30-second timeouts on a weka filesystem. A 120-second budget covers all observed cases.
